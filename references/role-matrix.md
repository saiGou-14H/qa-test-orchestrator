# Role Matrix

## Purpose

责任人/业务矩阵是运行态生成的工作面，不是静态岗位表。

不要先假设只有 `功能测试回归测试`、`自动化测试`、`测试开发SDET`、`CI质量平台` 四类目录。先从真实登录账号、菜单、审批节点、端入口和已跑通链路中识别当前项目实际存在的责任人和业务域，再决定如何拆任务、留证据和沉淀脚本。

测试执行时允许保留一层“测试岗位视角”，例如功能测试、回归测试、自动化测试、SDET；但这层用于任务拆分和覆盖规划，不用于替代动态责任人/业务域目录。

## How To Build The Matrix

### 1. Identify owners or roles

优先从这些证据识别责任人：

- 登录账号的实际归属
- 登录后可见菜单和待办入口
- 审批链或流转节点的角色名
- 页面中明确写出的端形态或设备形态
- 业务动作最终归属到谁执行

常见结果可能是：

- `管理员（教务）`
- `班主任`
- `教务主任`
- `分管领导`
- `主管领导`
- `学员端（移动端）`

这些只是说明样式，不是固定枚举。新项目里如果出现 `运营专员`、`教师端（H5）`、`审批秘书`，就直接按真实名称建矩阵。

### 2. Identify business domains under each role

优先从这些证据识别业务域：

- 菜单名、页签名、面包屑、页面标题
- 流程名、审批名、通知名、考试名、课程名
- 列表/详情/表单共同归属的模块
- 已跑通链路中可稳定复用的业务标签

常见结果可能是：

- `登录与权限`
- `办班申请与审批`
- `课表与课程`
- `学员与报到`
- `考勤与签到`
- `评价问卷`

如果源码里叫 `classApply`，但页面上写的是“办班申请”，优先使用“办班申请”这类用户可读名字。

### 3. Split work by role and domain

推荐把每个 `责任人 x 业务域` 视为一个最小工作包：

- 同一责任人下的多个业务域分开执行、分开留证。
- 同一业务域出现在多个责任人下时，各自保留独立路径，不混放。
- 同一责任人、同一业务域下如果存在多条独立链路，按链路拆成多个脚本或多个报告章节。

### 4. Add a testing-role view when needed

如果任务需要按测试岗位拆分，可以在矩阵上再叠加一层测试视角：

- 功能测试：关注主流程、异常分支、表单校验、权限切换、提示文案
- 回归测试：关注历史功能不被破坏、核心链路再次验证、缺陷复测
- 自动化测试：关注 E2E 主链路、可重复执行脚本、浏览器流程与接口串联
- SDET：关注最小回归集、测试数据构造、辅助脚本、稳定性与可维护性

这一层只用于任务分配、执行摘要和覆盖说明，不改变最终产物目录。最终截图和脚本仍然回到 `责任人 x 业务域` 的路径。

## Suggested Matrix Fields

至少记录下面这些字段：

- 责任人/角色
- 端形态或登录入口
- 可见菜单/待办入口
- 业务域
- 最小可跑通链路
- 测试岗位视角：功能测试 / 回归测试 / 自动化测试 / SDET
- 使用的账号与测试数据
- 证据路径
- 当前状态：未测 / 已测 / 阻塞 / 待复测

## Minimal Example

```text
责任人: 管理员（教务）
入口: PC 后台登录页
业务域: 课表与课程
链路: 打开课表管理 -> 导入课表 -> 预览 -> 确认导入
证据: artifacts/screenshots/管理员（教务）/课表与课程/
脚本: artifacts/automation/管理员（教务）/课表与课程/
```

```text
责任人: 学员端（移动端）
入口: H5 学员登录页
业务域: 评价问卷
链路: 进入评价列表 -> 打开答卷页 -> 提交答卷 -> 查看结果
证据: artifacts/screenshots/学员端（移动端）/评价问卷/
脚本: artifacts/automation/学员端（移动端）/评价问卷/
```

## Update Rules

- 运行中新增责任人或业务域时，立即把它补进矩阵和目录。
- 不要把新发现的对象强行塞进已有固定分类。
- 如果某条链路被证伪，回到最早不确定的节点重建矩阵，不要靠猜测补全。
