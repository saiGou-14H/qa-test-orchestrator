# Deliverables

## Artifact Layout

截图、调试图和复测脚本共享同一套动态 taxonomy：

```text
artifacts/
  screenshots/
    管理员（教务）/
      登录与权限/
        01-进入登录页.png
        02-填写账号密码.png
        03-登录成功-续测.png
      课表与课程/
        01-打开课表管理弹窗.png
    _debug/
      管理员（教务）/
        课表与课程/
          01-定位弹窗入口-探针.png
  automation/
    管理员（教务）/
      登录与权限/
        01-登录与首页校验.py
      课表与课程/
        01-课表导入回归.py
  report/
  manual/
  data/
```

## Screenshot Rules

- 正式截图放 `artifacts/screenshots/<责任人>/<业务域>/`
- 调试截图放 `artifacts/screenshots/_debug/<责任人>/<业务域>/`
- 目录名使用真实责任人和真实业务域，不使用固定岗位模板目录。
- 同一责任人下的不同业务域必须分目录；同一业务域在不同责任人下也必须分目录。
- 文件名使用 `步骤序号-动作描述[-状态].png`
- 文件名不要重复完整责任人和业务域，因为它们已经体现在目录层级里。
- 动作描述直接写当前链路动作，不用 `screenshot-1`、`final`、`debug` 这类无语义名字。
- 如果同一业务域里有多条链路，动作描述必须带上足以区分链路的业务关键词，避免重名。
- 允许的状态后缀优先使用：
  - `续测`
  - `复核`
  - `探针`
  - `回退验证`

## Naming Examples

- `artifacts/screenshots/班主任/办班申请与审批/01-进入办班申请页.png`
- `artifacts/screenshots/班主任/办班申请与审批/02-打开新增弹窗-续测.png`
- `artifacts/screenshots/管理员（教务）/课表与课程/03-确认导入成功-复核.png`
- `artifacts/screenshots/_debug/管理员（教务）/课表与课程/01-定位课表弹窗入口-探针.png`

样板中原本写在文件名前缀里的业务，例如 `办班申请与审批`、`课表与课程`、`评价问卷`，现在应进入业务子目录，而不是继续平铺在文件名前缀里。

## Report Rules

推荐在 `artifacts/report/` 下输出 Markdown 报告，按责任人和业务域组织章节。

推荐章节：

1. Outcome
2. Key evidence
3. Verification
4. Next step

每条业务验证建议包含：

- 责任人
- 业务域
- 链路目标
- 前置账号/数据
- 关键步骤
- 预期结果
- 实际结果
- 结论

示例：

```markdown
### 管理员（教务） / 课表与课程 / 课表导入回归

- 前置：管理员测试账号已可登录
- 链路：打开课表管理弹窗 -> 上传课表 -> 预览 -> 确认导入
- 预期：导入成功并可回显到列表
- 实际：导入成功，列表回显正常
- 结论：通过

![确认导入成功](../screenshots/管理员（教务）/课表与课程/03-确认导入成功-复核.png)
```

## User Manual Rules

- 用户手册只保留稳定成功路径，不混入缺陷现场和探针截图。
- 用户手册引用正式截图，不引用 `_debug` 下的内容。
- 用户手册沿用责任人和业务域的层级来组织章节，方便业务侧直接复用。
